# リージョンを変えずに表示を速くする（米国オリジン・日本からのアクセス）

AWS のリージョンが米国（us-east-1 など）のままだと、日本からアクセスしたときにレイテンシで表示が遅く感じることがあります。**リージョンはそのまま**で、次の方法で体感速度を改善できます。

---

## 1. フロント側の最適化（すでに取り入れているもの）

このリポジトリでは次の対策を入れています。

| 対策 | 内容 |
|------|------|
| **preconnect** | `index.html` で API ドメイン（例: `https://api.artist-motion.com`）へ先行接続。API 呼び出しの遅延を削減。 |
| **preload** | ヒーローで使う最初の画像を `<link rel="preload" as="image">` で先読み。 |
| **ヒーロー表示** | 4枚すべての読み込み完了を待たず、**1枚目が読めた時点で**ヒーローを表示（タイムアウト 2.5 秒）。残りはバックグラウンドでプリロード。 |
| **lazy loading** | お知らせ一覧などの画像に `loading="lazy"` を付与し、初期表示の転送量を減らす。 |

- **API を別ドメインで使う場合**: `client/index.html` の `preconnect` の URL を、実際の API のオリジン（例: `https://api.artist-motion.com`）に合わせてください。同一オリジン（`/api`）の場合は preconnect 行を削除して構いません。

---

## 2. 推奨: CDN（CloudFront）を前面に置く

**リージョンは米国のまま**で、**CloudFront などの CDN をオリジン（S3 / EC2 / Amplify など）の前に配置**すると、日本を含むエッジからキャッシュを配信できるため、初回以降の表示がかなり速くなります。

- **静的コンテンツ**（HTML, JS, CSS, 画像）: CloudFront でキャッシュし、日本など近いエッジから配信。
- **API**: キャッシュしない、またはニュース一覧など一部だけ短い TTL でキャッシュする設定も可能。

### 手順のイメージ（AWS の場合）

1. **CloudFront ディストリビューションを作成**
   - Origin: いま配信している元（S3 バケット / EC2 の Nginx / Amplify など）。
   - 必要に応じて「オリジンリクエストポリシー」「キャッシュポリシー」で、静的ファイルは長めの TTL、API はキャッシュなし（または短い TTL）に分ける。

2. **ドメインを CloudFront に向ける**
   - 本番ドメイン（例: `artist-motion.com`）の DNS を CloudFront のドメイン名（`xxxxx.cloudfront.net`）に向ける、またはカスタムドメイン＋ACM 証明書を CloudFront に設定。

3. **キャッシュヘッダー**
   - オリジン（Nginx や S3）で静的ファイルに `Cache-Control: public, max-age=31536000` など長めの max-age を付与すると、2回目以降の表示がさらに速くなります。

リージョンはそのままで、**「オリジンは米国 → 配信はエッジ」** の形にすると、日本からの体感速度が大きく改善します。

---

## 3. その他のヒント

- **画像の最適化**: WebP 化・適切な解像度にすると転送量が減り、表示が軽くなります。
- **API の応答**: ニュース一覧など、可能なら必要最小限のフィールドだけ返すようにすると初回表示が軽くなります。

---

まとめ: **リージョンを変えずに表示を速くするには、(1) 上記のフロント最適化を維持し、(2) CloudFront など CDN を前面に置いてキャッシュ配信する**のが最も効果的です。
